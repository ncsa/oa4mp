This descriubes testing OA4MP with the various authentication methods as part of the release cycle.
Alternate web.xml files reside in server-admin/src/main/resources/web-inf for various cases.

The basic idea is to copy the correct one to the oa4mp-server-oauth2/src/main/webapp/WEB-INF/web.xml file,
tweak the server configuration and run the tests.

1. Basic Authentication
 This uses built in authentication (username/password) and is the default. It does
 not use any other mechanism.


Web file:proxy-web.xml
Test case: Any stadnard client should work for testing auth code or device flow.

authorizationServlet is missing. Note that this requires a class that overrides

server-api/src/main/java/org/oa4mp/server/api/storage/servlet/AuthenticationUtil.java

checkUser to check for username and password. Normally this implies you hav written a system
that has these OR are fronting an existing user management system with OA4MP.

the proxy-web.xml file works because ti si the most minimal and does not have any other machiner.
The entire flow is handled by OA4MP itself.

2. Proxy

In this case, antoher OAuth2 server is used to proxy the requests. Testing uses CILogon.

Web file: proxy-web.xml
Test case: Any stadnard client should work for testing auth code or device flow.
server cfg: authorizationServlet use is "proxy" and typical entry is

        <authorizationServlet use="proxy"
                              cfgFile="/home/ncsa/dev/csd/config/client-oa2.xml"
                              cfgName="proxy-client"
                              localDFConsent="true"/>

3. Tomcat

This uses the built in Tomcat authentication. That means that there must be configured users in the
Tomcat user file and that the roles for them must be configured correctly. Really any system that sends
along the REMOTE_USER header works

Tomcat user file: tomcat-users.xml put in a role and make sure there is a user that has it, e.g.

  <role rolename="oa4mp"/>
  <user username="jeff" password="XXXXXXX" roles="tomcat,oa4mp"/>

Web file: header-web.xml
Test case: Any stadnard client should work for testing auth code or device flow.
server cfg: authorizationServlet useHeader is true and typical entry is

        <authorizationServlet useHeader="true"
                              requireHeader="true"
                              headerFieldName="REMOTE_USER"/>

This sets the header field name.  Note that the web.xml file has all the machinery in it to enable
Tomcat logins for this user.
