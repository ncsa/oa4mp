Introduction

A virtual organization in OA4MP is an abstract model of a dedicated organization's issuer. It provides unique signing keys, an issuer, well-known endpoint and other essentials.

Who uses them?

The standard way is that an administrative client joins the VO. At that point, every standard client that it manages is in the VO and will have all the necessary changes done to its tokens automatically. Note that if yo have existing clients and add them to a VO, they must be made aware of it as well as an consumers of their tokens, since the issuer will change.

Creating a VO

This is done with the CLI. Use the vo component. Issue a create command and follow the instructions. There are a few items to note:

  • You may set the id token issuer as the issuer property and the issuer asserted in access tokens with the at_issuer. Frequently these are not the same.
  • This will generate a set of keys for you if you want or it will use a pre-existing set. If you want to use a pre-existing set, make sure that they exist. You will be prompted for a path to them.
  • The id for the VO is vo_id. In the admin client there are properties for vo and vo_uri. In an earlier version, these were different. They should always be the same and identical to the vo_id.

Inside the vo component, help is available. Use it! So you may issue /help at any time and get a list of all the properties. This facility is also available during object creation. This means that as you are prompted for each property, you can issue /help as the value and the help for that will be printed. E.g. if you are prompted for the name you could issue 

 enter the name[(null)]:/help
 Property: String. (--)
 The human-readable name for this client. This is displayed on the consent
 screen, so it really should make sense to the user and they know exactly what they are
 consenting too.
 enter the name[(null)]:

So the help is printed, then you are prompted again for the value of the name.
